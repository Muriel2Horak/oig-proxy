- Twin MQTT ingestion can reuse the existing `mqtt_publisher.add_message_handler()` thread-to-async bridge pattern (`run_coroutine_threadsafe`) to keep callback behavior consistent with ControlPipeline.
- For twin queueing over MQTT, topic-derived routing (`oig_local/<tbl_name>/<tbl_item>/set`) plus payload overrides provides a stable DTO mapping while preserving optional fields (`request_key`, `received_at`).
- Invalid payload handling should explicitly catch both UTF-8 decode failures and `json.JSONDecodeError`, log at ERROR level, and return early to prevent crashes in MQTT callback flow.
- Proxy-side integration should initialize MQTT subscriptions after loop attachment and MQTT connect startup, replacing direct ControlPipeline MQTT setup with `TwinMQTTHandler.setup_mqtt()` when twin is enabled.
- Twin state publishing fits best as a retained raw MQTT payload (`oig_local/oig_proxy/twin_state/state`) generated from in-memory queue/inflight/session fields and triggered directly in each mutating DigitalTwin method.
- SA follow-up commands can be safely auto-enqueued from `on_tbl_event()` only after an `applied` transition, using a non-SA guard (`tbl_box_prms/SA`) plus an `auto_generated=True` marker on the queued DTO to avoid recursive loops and preserve traceability.
- Session-scoped twin routing in ONLINE mode can be implemented safely by arming a `_pending_twin_activation` flag from `TwinMQTTHandler.on_mqtt_message()`, then enabling `_twin_mode_active` only on the next BOX connection when `DigitalTwin.get_queue_length()` confirms queued work.
- Local-control routing should treat ONLINE mode as cloud-by-default and twin-only when `_twin_mode_active` is set; when routing resolves to twin, `IsNewSet` polls must never fall through to cloud forwarding.
- Legacy `pending_frame` staging can be fully removed once Control API writes are routed through `ControlSettings.queue_setting()` -> `DigitalTwin.queue_setting()`, because poll-time frame delivery is already owned by `DigitalTwin.on_poll()`.
- Offline fallback in proxy/cloud forwarding can be simplified to a pure local ACK responder (`_respond_local_offline`) with no setting-delivery branch; this eliminates duplicate legacy delivery paths and stale-connection risks.
- Removing production imports of `control_pipeline` requires replacing shared helpers (format/coerce) locally (`proxy_status`, `mqtt_state_cache`) so the legacy module can remain only as a compatibility shim.
- HYBRID offline should route setting polls by `in_offline` state (not by `should_try_cloud()`), so retry probes can run for data frames while `IsNewSet/IsNewWeather/IsNewFW` stay on twin delivery.
- Removing `CloudForwarder.handle_frame_offline_mode()` keeps offline ACK behavior in proxy-level handling and prevents accidental bypass of twin-first poll routing.
- A dedicated `DigitalTwin.handle_frame(frame, conn_id)` entrypoint can preserve queue/retry isolation by routing only `IsNewSet` to `on_poll()` while treating other frames as stateless flow: optional cloud forward + unconditional MQTT publish.
